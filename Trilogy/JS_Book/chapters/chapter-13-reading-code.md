# Chapter 13 — Reading Other People's Code

You will spend more of your career reading code than writing it. Reading the codebase you joined a week ago. Reading documentation. Reading library source to figure out why something doesn't work the way you expected. Reading Stack Overflow answers. Reading code generated by AI to verify it does what it claims.

Reading code is a skill. It's a different skill from writing. Most beginners are decent at writing — they have tutorials and exercises that teach it. Most beginners are bad at reading — nobody teaches it explicitly. They land in their first real codebase and feel illiterate, even though they wrote a hundred lines of clean code yesterday.

This chapter is about closing that gap. It's the chapter I wish I'd had at the start of my career.

---

## The mistake beginners make

The mistake is trying to read code line by line, top to bottom, the way you'd read a novel. Code isn't structured for that. A function in line 50 might be called from line 200. A constant in line 10 might be used in twelve places throughout the file. Reading sequentially gives you the structure of the file but not its meaning.

The right way is to read code *outside-in.* Start at the top level — what does this module export? What does this script do on load? What functions are called from where? Build a map of the high-level shape before you dive into any individual function.

---

## Where to start

Open a JavaScript file you didn't write. Maybe an open-source library. Maybe the source of a small site you like. Maybe the field guide's script after a couple of weeks away from it.

Don't read it line by line. Instead:

**1. Look at the imports and exports.** The imports tell you what the file depends on. The exports tell you what it provides to others. A file that imports `fetch` and exports `getUser` is, at a glance, a file that fetches users. You don't need to read the body to know that.

**2. Look at the top-level statements.** Is the file mostly function declarations? Or does it run code on load? A file with only declarations is a library — it provides functions for others to call. A file that runs code on load is an entry point — it does something when invoked.

**3. Skim the function names.** Functions are named (in clean code) for what they do. A glance at the names tells you the file's vocabulary: `loadUser`, `formatName`, `validateEmail`, `saveProfile`. You now know roughly what kinds of operations the file handles.

**4. Pick one function and read its body.** Now you're reading code, with context. The body refers to variables and other functions; you know what they're for because of the names.

This sequence — imports, top-level, names, then bodies — gives you a working model of the file in five minutes, instead of an hour spent confused.

## When you don't understand a piece

When you hit something you don't understand, the question is: *do I need to understand it right now?*

Often the answer is no. You're trying to figure out *what the file does,* not *every detail.* If the function name is `parseISODate` and the body is a regex you don't recognize, you don't need to understand the regex to know what the function does. You can move on.

Make a note ("come back to: parseISODate's regex") and continue. The mental model survives. The detail can be filled in later if it matters.

If the answer is yes — you genuinely need to understand this — break the problem down. Look up unfamiliar syntax on MDN. Run snippets in the Console. Add `console.log` calls and read what they output. Treat the code as a thing you can experiment with, not a thing you have to understand by reading alone.

---

## Reading library code

Open-source library source is one of the best ways to learn JavaScript. The code is professional, often well-commented, and you can see how experienced developers solve real problems.

Start with libraries you already use, and pick something small. A utility library like *date-fns* has functions that are mostly self-contained — pick one, find its source on GitHub, read it. Most are 5–20 lines of well-named code. You'll learn idioms, patterns, and edge cases you wouldn't encounter on your own.

Don't start with React or Express or anything large. Big frameworks have layers of abstraction that take years to understand. Small libraries have flat code that teaches more per page read.

---

## Reading code generated by AI

A new skill: verifying that AI-generated code does what it claims. AI can produce code that looks correct, runs without errors, and is subtly wrong. The same outside-in approach works:

- Look at the imports the AI used. Are they real? (AI sometimes invents libraries that don't exist.)
- Look at the function shape. Does the signature match what you asked for?
- Read the function body. Does the logic match what you'd expect?
- Run it. Test edge cases. AI is often confident about cases it didn't handle.

Don't take generated code on faith. Read it the way you'd read a junior developer's pull request — assume nothing, verify everything. The same pattern recognition you build by reading library code makes you better at reading AI-generated code.

## A small honest moment

For my first two years of writing code, I would open someone else's project, scroll through it, get overwhelmed, close the tab, and decide my project was too small to need that kind of complexity. I never read other people's code. I learned everything in isolation.

The result was that I rebuilt patterns from scratch that the community had solved years before. My code was idiosyncratic — solid in places, naive in others — because I had no reference for "what does professional code look like."

The fix was forcing myself to read one open-source library a week. Just one. Just one file. Twenty minutes. After three months, my own code was meaningfully better, and I had stopped feeling intimidated by codebases I didn't write.

## *Reading code is a skill. It is not the same skill as writing code, and you will use it more.* <!-- SIGNATURE LINE -->

Practice it deliberately. Open files you didn't write. Spend twenty minutes a week. Build the habit. By the time you're working on real projects, the habit will be there, and you'll move through unfamiliar codebases the way you move through your own.

---

## A small permission

You will not understand everything you read. That's fine. The goal of reading isn't comprehension — it's *familiarity.* Each pass, you understand a little more. Each library you read, the next one is easier. The skill compounds.

Beginners think understanding has to be complete to count. It doesn't. Partial understanding, accumulated over months, becomes complete understanding. Read more than you think you should.

---

## Try this

1. Open the source of a small JS library you've heard of — `is-odd`, `lodash.debounce`, anything small. Find it on GitHub. Read the source.
2. Spend 20 minutes. Don't aim for full understanding. Just look at imports, structure, function names, then dive into one function.
3. Note one thing you learned that you didn't know.
4. Open the source of one of your own modules from earlier in this book. Read it the same way. Notice what's clear and what's not.
5. Commit any cleanups you want to make to your own code based on what you read:
   ```
   git add .
   git commit -m "Code cleanup after reading session"
   ```
